# analyzer — 代码分析 Skill

一个帮助便捷分析代码的 AI skill。它引导分析过程，并把每一轮分析沉淀为一份结构化、持续增长的**分析报告**。

## 这个 skill 做什么

- **产物是报告**：分析的最终交付物永远是当前工作目录下的一个 Markdown 报告文件，对话只是过程。
- **先锁定代码身份**：每次分析先记录仓库（`git remote -v`）、分支 / tag / commit，写进报告开头。
- **范围可大可小**：支持整个仓库、某个模块/目录，或某个模块的某个具体方面（如并发、安全、性能、错误处理）。
- **多轮问答累积**：一次分析常跨多轮交互，每轮的问题与回答都会被提炼并追加到报告的「分析日志」中。
- **图示优先用 mermaid**：调用图、流程图、时序图、状态机、依赖图等优先用 mermaid 渲染成图；逐行注释的密集调用追踪用字符树作为补充。
- **不破坏工作树**：当请求的分支/tag 与当前 checkout 不匹配时，绝不切换分支，而是用 `git show` / `git ls-tree` / `git diff` 等只读方式读取目标代码。

## 目录结构

```
analyzer/
├── SKILL.md                      # 主入口：触发条件、工作流、操作条款
└── references/
    ├── report-template.md        # 分析报告完整模板
    ├── mermaid-cheatsheet.md      # mermaid 图示速查（图类型选择 + 示例）
    └── git-cheatsheet.md         # 非破坏性 git 命令速查
```

## 报告长什么样

```markdown
# 分析报告：<仓库名> — <分析范围 / 主题>

## 元信息
- 仓库 / 请求的 ref / 实际分析的 ref / 分析范围 / 关注方面 / 日期

## 摘要 / 关键发现

## 分析日志
### Q1：<问题>
<带 文件:行号 引用和代码片段的回答>
```

报告文件命名：`analysis-report-<仓库名>[-<范围>]-<ref>.md`，例如
`analysis-report-myrepo-scheduler-concurrency-v1.2.0.md`。同一分析范围跨轮追加到同一文件，不同范围各自独立成报告。

## 怎么用

把这个目录放到你的 skills 目录下（安装方式由你自行处理）。之后在对话中提出分析需求即可触发，例如：

- 「分析一下这个仓库的整体架构」
- 「帮我看下 `pkg/scheduler` 模块的并发安全，基于 tag v1.2.0」
- 「认证模块的错误处理有没有问题」

skill 会确定目标与范围、创建/定位报告，并在每轮交互后把问答写入报告。

## 核心规则（摘自 SKILL.md）

1. 绝不为满足 ref 请求而改动工作树。
2. 报告即产物——问答未写入报告，这一轮分析不算完成。
3. 报告开头必须是仓库 + ref 元信息。
4. 一次分析范围 = 一个报告文件，跨交互追加。
5. 请求的 ref 与实际分析的 ref 不同时，必须区分标注。
6. 范围要显式且聚焦，写进标题与元信息。
7. 证据优先于断言，用 `文件:行号` 支撑发现。
8. 不臆造，不确定的内容如实标注。
