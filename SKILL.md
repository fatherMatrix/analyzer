# 代码分析 Skill（analyzer）

帮助用户分析代码，并把分析过程沉淀为一份持续增长、存放在当前工作目录的报告文件。

**最终交付物永远是一个报告文件。** 对话是临时的，报告才是产物。用户提出的每一个有价值的问题、你给出的每一个回答，都必须写进报告。

## 核心工作流

### 1. 先确定分析目标（每次都先做）

在分析任何代码之前，先弄清楚你看的到底是*哪一份*代码。

```bash
git remote -v                      # 仓库位置
git rev-parse --abbrev-ref HEAD    # 当前分支
git describe --tags --always       # 当前 tag / commit
git rev-parse HEAD                 # 当前 commit 完整哈希
```

如果用户指定了分支、tag 或 commit：

- 把它和当前 checkout 对比（`git rev-parse --abbrev-ref HEAD`、`git describe --tags`）。
- **如果不匹配，禁止切换分支、checkout 或以任何方式改动工作树。** 用户可能有未提交的改动或保持当前状态的其他原因。应当用只读方式读取目标代码：

  ```bash
  git show <ref>:<路径>              # 读取某分支/tag/commit 下的文件内容
  git ls-tree -r --name-only <ref>   # 列出某 ref 下的所有文件
  git log <ref> --oneline -20        # 查看某 ref 的历史
  git diff <refA> <refB> -- <路径>   # 跨 ref 对比
  ```

  其中 `<ref>` 可以是分支名、tag 或 commit 哈希。

- 在报告中记录：用户**请求的 ref** 与当前**实际 checkout 的 ref**，并注明本次分析是通过 `git show`/`git ls-tree` 读取目标 ref 完成的，工作树未被改动。

如果当前目录不是 git 仓库，在报告中注明，并按普通文件分析继续（仓库/分支一节写「非 git 仓库」）。

### 2. 明确分析范围

分析的范围可大可小，先和需求对齐，再开始动手。范围通常是以下三档之一：

- **整个仓库**——架构、整体设计、技术栈概览。
- **某个模块 / 子系统 / 目录**——例如 `pkg/scheduler`、某个服务、某个包。
- **某个模块的某个具体方面**——例如「`scheduler` 的并发安全」「认证模块的错误处理」「某接口的性能瓶颈」。这是最常见、也最有价值的一档。

要点：

- 从用户表述中提炼出范围三要素：**范围层级**（仓库 / 模块 / 方面）、**目标路径**（哪个目录或文件集合）、**关注方面**（若有，如并发、安全、性能、错误处理、可维护性等）。
- 如果范围模糊（例如只说「帮我看下这个模块」却没说看哪方面），可以先按合理默认推进，并在报告中写明你采用的范围；或在一两句内向用户确认。
- 范围决定你读多少代码：**不要因为范围是「某模块的某方面」就去通读整个仓库**，聚焦目标路径与相关调用链即可。
- 把范围写进报告的标题和元信息（见下），让读者一眼看出这份报告覆盖什么、不覆盖什么。
- 范围在多轮交互中可能扩大或收窄——发生变化时更新报告的「范围」一节，并在分析日志里记一笔。

### 3. 创建或定位报告

- 报告存放在**当前工作目录**。
- 命名：`analysis-report-<仓库名>[-<范围>]-<ref>.md`（仓库名、范围、ref 均清理为文件系统安全的 ASCII 字符；范围段可选，用于区分针对不同模块/方面的报告，例如 `analysis-report-myrepo-scheduler-concurrency-v1.2.0.md`）。如果当前目录已有范围匹配的报告，**应追加而非新建**——同一分析范围就是一份报告，哪怕跨越多轮交互。
- 新建报告前，先列出当前目录中已有的 `analysis-report-*.md` 文件，复用范围匹配的那一份；范围不同（不同模块或不同关注方面）则新建独立报告。

### 4. 写报告头部（创建报告时写一次）

精确结构见 `references/report-template.md`。头部至少包含：

- 一个有描述性的**标题**，应体现分析范围（例如 `# 分析报告：<仓库> — scheduler 模块并发安全`）。
- **仓库**（`git remote -v` 输出，或 origin 地址）。
- 被分析的**分支 / tag / commit**（如有差异，同时记录「请求的」和「实际的」）。
- **分析范围**：范围层级、目标路径、关注方面。
- 分析日期。

### 5. 分析，并记录每一轮问答

这是最重要的规则：**一次分析通常是多轮对话。** 每一轮通常包含一个问题（显式或隐含）和你的回答。在对话中回答后，要把它提炼出来**追加**到报告的问答日志中。

- 从用户的话中提取出真正的问题，即使表述模糊也要在报告里清晰地复述出来。
- 记录你的回答：发现、相关的 `文件:行号` 引用、代码片段、结论。
- 在 `## 分析日志` 一节中按时间顺序排列，每条形如 `### Q<n>：<问题>`，后接回答。
- 如果用户一条消息里包含多个不同问题，拆成多个编号条目。
- 已有回答只在纠错时修改；否则只追加新条目，不改写历史。

### 6. 让报告自洽、可读

- 代码引用统一用 `路径/文件.扩展名:行号`，便于溯源。
- 只引用*相关*的几行代码，不要贴整个文件。
- 优先给出具体证据（文件、函数、行号），而非笼统总结。
- 在报告靠前处维护一节 `## 摘要 / 关键发现`，随着理解加深持续更新。
- **图示优先用 mermaid**（流程图 / 调用图 / 时序图 / 状态机 / 依赖图），让读者先看到清晰的逻辑框架；需要逐行 `//` 注释的密集调用追踪可用字符缩进树作为补充。图类型选择与示例见 `references/mermaid-cheatsheet.md`。

## 报告骨架（速查）

```markdown
# 分析报告：<仓库名> — <范围 / 主题>

## 元信息
- 仓库：<git remote -v 的 origin 地址>
- 请求的 ref：<用户要求的分支/tag/commit，或「当前工作树」>
- 实际分析的 ref：<实际分析的分支 + tag + commit 哈希>
- 分析时工作树所在分支：<当前分支>（若与分析 ref 不同需注明）
- 分析范围：<整个仓库 / 模块路径 / 模块的某方面>（关注方面：<如并发、安全、性能…，无则写「整体」>）
- 日期：<YYYY-MM-DD>

## 摘要 / 关键发现
- ...

## 分析日志
### Q1：<问题>
<带 文件:行号 引用和代码片段的回答>

### Q2：<问题>
<回答>
```

## 条款 / 操作规则

以下是本 skill 的硬性规则：

1. **绝不为满足 ref 请求而改动工作树。** 不用 `git checkout`、`git switch`、`git reset`、`git stash`、`git clean`。用 `git show` / `git ls-tree` / `git diff` 读取其他 ref。
2. **报告即产物。** 未把对应问答写入报告文件前，这一轮分析不算完成。
3. **报告开头必须是仓库 + ref 元信息。** 如果无法确定仓库或 ref，在该节中明确说明，而不是省略。
4. **一次分析 = 当前目录中的一个报告文件。** 跨交互追加，不要把发现散落到多个文件。
5. **请求的 ref 与实际分析的 ref 不同时，必须区分标注**，让读者清楚到底看的是什么。
6. **范围要显式且聚焦。** 分析范围（仓库 / 模块 / 模块的某方面）必须写进标题与元信息；按范围聚焦读取相关代码，不为窄范围去通读整库。不同范围用不同报告文件。
7. **证据优先于断言。** 用 `文件:行号` 和代码片段支撑发现。
8. **不臆造。** 不确定或未经验证的内容，在报告里如实标注。

更完整的模板见 `references/report-template.md`；非破坏性 git 命令见 `references/git-cheatsheet.md`；mermaid 图示用法见 `references/mermaid-cheatsheet.md`。

## 注意点

1. 对于某些markdown阅读器，其表格需要前后有一个空行，否则无法渲染为表格。因此在输出中包含表格时，记得前后加一个空行
2. mermaid 图示必须写在独立的 ` ```mermaid ` 围栏里，且围栏前后各留一个空行。部分渲染器不支持 mermaid、会原样显示源码，因此关键图应在附近保留一句文字概述或字符调用树以便降级阅读；详见 `references/mermaid-cheatsheet.md`。
